4. おかしな箇所（cross-doc 整合ドリフト・未決）

D1. heartbeat/watchdog 停止の層割当が docs 間で不一致。
  • box-map / eval-gate 群は firmware(L0) に割当済: 自作境界に heartbeat / watchdog productization/06:225、H-G6「heartbeat lost → motor stop（上位通信非依存）・owner=firmware」productization/08:105、reason_code heartbeat_lost 08:113 / 06:228 / mode-x-er/05:36,357。
  • 安全 canon / firmware は未反映: doc12 の Layer 0 は速度クランプ＋近接停止のみ 12:76-78、main.cpp:147-158 は「MCU に置く hard-cut は Phase 1 で判断」と明示先送り、PHASE1_CHECKLIST に heartbeat 項目なし。
  • なぜ重要（安全論点）: 通信断（ESP32↔Jetson リンク喪失・MCU 生存）時、最後の cmd_vel が residual で走り続ける runaway 経路。off-MCU の Guardian(L1) は死んだリンク越しに停止を押せない → 通信断で確実に止められるのは MCU 常駐 stop のみ。
  • 提案（所有境界を守る）: firmware は H-G6 に沿って Phase 1 で heartbeat/watchdog stop を実装（PHASE1_CHECKLIST に項目追加＝本 PR）。doc12 §Layer 0 への反映は safety-state 所有 → decision-needed として送る（本 PR では doc12 を書き換えない）。
D2. R-43 LaserScan MTU 方針が未決。 360°/0.4°≈3.6KB/scan が UDP MTU 512B 超過（2台常時送出）doc07:253。選択肢＝scan ダウンサンプル(0.4°→1–2°) / Reliable・MTU 設定 / 最悪 USB 有線。どれを既定にするか Phase 1 実機（T3 併せ）で確定。R-37 host spike はこの MTU を未検証 spike/RESULT.md。
  • 実測で確定する定数（発明しない・引数扱い）: TRACK_WIDTH・速度→PWM duty 曲線・encoder scale kinematics.h / PHASE1:31,38、MAX_ANGULAR_VELOCITY（現 2.0=placeholder config.h:10）、ロボット実寸 PHASE1:46。
  • 秘密: WiFi SSID/PASS・Agent IP は config_secret.h（gitignore・コミット禁止 .claude/rules/safety.md）。
