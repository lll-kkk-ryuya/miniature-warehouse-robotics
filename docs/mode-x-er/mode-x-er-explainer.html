0. 位置づけ（Mode X-ER とは / Mode 名）

何をするモードか
Gemini Robotics-ER を「音声・画像・状態から倉庫タスクを理解する司令塔」にする。既存 Mode A/B/C の LLM provider 比較とは分け、Gemini Robotics-ER 固定で音声指示の解釈・俯瞰画像の object target 認識・task graph 分解・既存安全経路への接続を検証する。README:9-14
ER に閉じ込める範囲
ER は 「見る・理解する・計画する」層に閉じ込め、Nav2 / ROS / Jetson / /cmd_vel を直接叩かせない。実行可能 command 候補への変換は L3 Robotics Planning Core が担う。README:16 / 01:108-130

3. L3 Planning Core 詳細（4 stage）

ER adapter の出力形式が変わっても L3 以降が大きく変わらないよう 4 段に分ける。各段は型付き中間生成物を産む typed normalizer。02 §目的 9-19

Known Location snap（MVP の肝）: known location は倉庫マップ上の名前付き位置（shelf_1 / charging_station / retreat_A 等）で、赤箱/青箱そのものではない。location key は凍結契約 warehouse_interfaces.locations.KNOWN_LOCATIONS と config/warehouse.base.yaml で同期。CommandItem.destination は known location だけを許すため、MVP は visual target を known location へ snap できた場合だけ compile する。coordinate goal は未凍結＝compile しない。02 §Known Location 21-37 / locations.py KNOWN_LOCATIONS
① Validator — 実行候補として扱えるかを actuation 前に構造化判定（JSON 変換ではない）。結果は accepted / reject / needs_clarification の3値（confidence・clarification は operator clarification へ戻す）。accepted 以外は 0 dispatch。02 §1 Validator 39-84
検証カテゴリ: parse / schema / robot registry / action allowlist / target reference(detections[].id または known location) / confidence(→ reject or needs_clarification) / graph reference(after) / graph structure(DAG) / state freshness / emergency state / clarification。02:72-84
拡張設計（商用で最重要の拡張点）: Pydantic=syntax/type/required 正規化・JSON Schema=fixture/外部 UI/contract 説明・custom rule は PlanPolicy として分離（model adapter に埋めない）。rule result=code / severity / field_path / message_for_operator / debug_detail / dispatch_effect。stable code（UNKNOWN_ROBOT/UNKNOWN_ACTION/UNKNOWN_TARGET/LOW_CONFIDENCE_TARGET/INVALID_AFTER_REFERENCE/TASK_GRAPH_CYCLE/CYCLE_STATE_STALE/EMERGENCY_ACTIVE）。policy は project default → site profile → runtime safety state。threshold は docs/config/contract まで hardcode しない。02:86-99
無いと: 存在しない bot3 / 未知 action が下流到達・低 confidence 誤検出が robot 移動になる・古い state の command 発行・emergency 中に新 motion・商用で reject 理由を説明できない。02:101-107
▼
② Visual Resolver — 画像上の object target を robot が使える map target へ変換。pixel(u,v) → camera calibration/homography → map(x,y) → valid polygon check → known location snap → ResolvedTarget。map 外 / valid polygon 外 / reprojection error 過大は unresolved。ER の detection confidence と homography/snap confidence を合成して最終 confidence を作る。02 §2 Visual Resolver 109-159
calibration artifact = camera id / map frame / homography matrix / reprojection error / valid polygon。単一・固定の俯瞰カメラ前提で file として version 管理し現場ごとに差し替え（コード定数に埋めない）。02:148-158
▼
③ Task Graph Executor — task_graph を一度に全 dispatch せず、依存を守って ready task だけ出す。状態 pending → ready → running → succeeded / failed / cancelled。初回 cycle は t1 だけ ready、t2 は t1.completed（/nav2_bridge/goal_result / State Cache / RMF task state で確認）まで pending。DAG 検証・topological order は NetworkX 候補だが、runtime state machine は自作（NetworkX を wire/audit 正本にしない）。最初は process memory、商用は durable TaskGraphStore へ差替。02 §3 Task Graph Executor 161-198
無いと: 順序条件（bot1 到達後に bot2）無視・t1/t2 同時 dispatch・同一 task の二重 dispatch・failed/timeout/cancelled の扱い曖昧・UI/audit で task lifecycle 説明不能。02:186-192
▼
④ Command Compiler — ready task を既存の安全実行契約へ落とす最終段。output は凍結 warehouse_interfaces.schemas.Command に通す。禁止: gen_id/idempotency_key を作らない・known location 以外の coordinate goal を compile しない・ER の route/velocity/low-level action を採用しない・速度を絶対 compile しない。X-lite / X-rmf 差分は ExecutionProfile で分け、plugin 化（WarehouseNavCompiler / RmfTaskCompiler / ArmManipulationCompiler）。02 §4 Command Compiler 200-242
L3 で避けること: Gemini Robotics-ER の response shape を下流 MCP / Nav2 に直接漏らす / OpenCV の pixel coordinate をそのまま Command に入れる / NetworkX graph object を audit・wire schema に保存する / customer site 固有の camera calibration をコード定数に埋める / RoboticsPlan schema を最初から warehouse_interfaces に凍結する。02 §L3 Interface Skeleton 272-278
