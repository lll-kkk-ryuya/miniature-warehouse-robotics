4. 典型フロー — 「赤箱→赤ボール把持→トレー配置」

指示「赤い箱へ行って、赤い箱の中の赤いボールを掴んで、トレーに置いて」を、ER の task graph → L3 の順序実行 → VLA subtask で処理する。04:50-106

ER が作る高レベル task graph（移動系は command_action、操作系は内部 skill）:

t1 navigation command_action=navigate target=red_box_area t2 manipulation skill=grasp target=red_ball container=red_box after=t1.completed t3 navigation command_action=navigate target=tray_area after=t2.completed t4 manipulation skill=place target=tray object=red_ball after=t3.completed
04:60-68 ／ command_action=既存 Command 移動系のみ・skill=内部 manipulation subtask（既存 Command 契約へそのまま入れない）04:71-74
  1. Validator — robot / kind / command_action / skill / target / graph / state freshness / emergency を検証。
  2. Visual Resolver — red_box_area / tray_area を known location へ解決。未凍結の coordinate goal は compile しない。
  3. Task Graph Executor — 初回 cycle は t1 だけ ready、t2/t3/t4 は pending。
  4. Command Compiler — t1 を既存 Command(navigate) へ compile。
  5. L2 / L1 — MCP / Policy Gate / Nav2 が t1 を実行。
  6. Completion 判定 — Nav2 result・State Cache pose・task timeout・emergency state を見て t1.completed / t1.failed。
  7. VLA subtask 起動 — t1.completed 後、L3 が t2 を ready にし、OpenVLA へ限定 subtaskを渡す。
  8. Manipulation result — grasp 成功なら t2.completed。失敗なら L3 が retry / abort / ER replan を選ぶ。
到着判定は ER ではなく L3（04:108-109）: ER が「赤箱に到着した」と主観的に判断して OpenVLA を呼ぶのではない。到着判定は L3 が State Cache / Nav2 result / safety state を使って行い、その後で VLA subtask を起動する。

5. 統合 architecture の 3 オプション

ER と VLA の output をどう組み合わせるかで 3 案。初期は C → A → B の順に進める。01:17-109

A ER Primary + VLA Cross-Check

ER が RoboticsPlan draft を作り、VLA は visual grounding / target confidence の cross-check に使う。

＋ X-ER 設計を壊しにくい / 安全な cross-check

－ VLA の action 能力を使わない / disagreement policy が必要

01:17-47

B ER Planner + VLA L3 Candidate

ER が high-level task graph、VLA が Visual Resolver / action candidate を補助。

＋ 役割分担が明確 / L3 一部代行のユーザー意図に近い

－ Safety Compiler 設計が必要 / Command へ落とす境界が難

01:49-73

C Sim-First Policy 評価

実機 command path 接続前に Isaac Sim / offline fixture で policy candidate として評価。

＋ 実機前に相互作用を観察 / runtime・GPU・I-O を検証 / 失敗を golden fixture 化

－ 実機 E2E まで遠い / sim scene 整備が前提

01:75-97
① C: sim/offline 観察▶ ② A: grounding 補助なら▶ ③ B: action candidate なら 01:99-109
どの option でも ER/VLA から直接 Nav2 / ROS / Jetson / ESP32 を叩かせない。移動のみの scenario は Mode X-ER で先に完結させ、Mode X-ER-VLA では manipulation subtask を含む scenario を追加評価する（01:106-109）。

9. Simulation / Safety Gates（G0→G7）

ER と VLA の両方が候補を出すため、実機接続前に simulation と offline fixture を必須にする。03:11-22

G0
fixture — recorded image + text instruction + fake state を ER/VLA に入力 → response を保存できる
G1
output classification — grounding / action / trajectory / unknown のどれか → docs に output shape を記録
G2
fusion decision — ER と VLA の output 統合方式 → Cross-check / L3 candidate / Safety Compiler を選ぶ
G3
offline validator — invalid robot/action/target/stale/emergency を 0 dispatch → reject reason を ValidationReport 相当で残す
G4
sim-only replay — Isaac Sim / offline evaluator で candidate を replay → 実機なしで failure を再現
G5
compiler decision — Command Compiler で足りるか SafetyCompiler が要るか → docs に判断を残す
G6
MCP path — command candidate が MCP / Policy Gate を通る → bypass なし
G7
robot-gated — 実機接続を 人間 gate に → Jetson / ESP32 / safety の確認後のみ
実機接続前に禁止（03:24-30）: VLA velocity を /cmd_vel 直 publish ・ VLA trajectory を Nav2 action 直送 ・ Jetson service/REST を ER/VLA に教える ・ ESP32 速度 clamp を前提に上位安全を省略 ・ fixture/sim で未検証の output shape を実機に流す。

10. OpenVLA 採用判断 — task 別

移動だけの task は Mode X-ER（ER → L3 → Command → MCP/Policy Gate → Nav2）。OpenVLA は Nav2 だけで表現できない局所操作が入った時の候補。README.md:60-68 / 04:15-22,296-320

task推奨構成OpenVLA の扱い
赤箱へ移動するER + L3 + MCP/Policy Gate + Nav2不要
赤箱を視覚で見つけて移動するER + L3 Visual Resolver + Nav2原則不要・cross-check のみ候補
赤箱へ行って赤ボールを確認するER + L3 + optional VLA grounding補助として候補
赤箱の赤ボールを掴むER + L3 + VLA + SafetyCompiler + skill controller必要になる可能性が高い
赤ボールをトレーへ置くER + L3 + VLA + SafetyCompiler + skill controller必要になる可能性が高い
充電器へ戻る / 棚Aへ行くER + L3 + Nav2不要

✅ OpenVLA を使う（04:310-319）

  • 移動後に grasp / place / align / dock など局所操作が必要
  • 対象物の位置・姿勢が毎回変わる
  • output を grounding_report / action_candidate に分類できる
  • SafetyCompiler 等で reject できる ・ fixture/sim gate 通過

❌ OpenVLA を使わない（04:296-308）

  • named location への移動だけで完結 / Nav2・Open-RMF で解ける
  • target が known location に snap でき把持/配置が不要
  • runtime / license / GPU 要件が未確認
  • output class を説明できない / reject path 未確認
