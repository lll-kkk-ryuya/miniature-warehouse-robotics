# システム設計 Q&A

作成日: 2026-05-22

本ドキュメントは、プロジェクト設計時のQ&Aを体系的にまとめたものである。各セクションは「疑問→回答→結論」の構造で記述する。

---

## 1. 全体構成 — 各機器の役割と指示の流れ

### Q: Claude（LLM）はどのような判断を行うのか？

Claude（LLM）は**戦略レイヤーのみ**を担当する。

| 判断レベル | 担当 | 応答速度 | 具体例 |
|-----------|------|---------|--------|
| 戦略（何をやるか） | LLM（Claude等） | 1-3秒 | 「Bot1にtask_1、Bot2にtask_2を割当」 |
| 交通管理（鉢合わせない） | Open-RMF | 即時 | 「Bot2は5秒待機、Bot1が先に通路A通過」 |
| 反射（ぶつからない） | Nav2 DWB/MPPI | 50ms | 「目の前に壁、停止」 |

LLMが行う判断（モードC: Open-RMF主方針）:
- タスク割当（どのロボットがどこへ行くか）
- 優先順位決定（安全性 > 効率性）
- バッテリー管理（3段階: 10%以下は緊急停止、10-20%は新規タスク禁止+充電推奨、20-30%は充電候補）
- エスカレーション対応（Open-RMFが3回失敗した場合のみ）

※交通管理（渋滞回避・迂回・待機指示）はOpen-RMFが即時処理する。モードA/BではClaude自身が交通管理も行う（YouTube比較検証用）。

LLMが**行わない**こと:
- リアルタイムのモーター制御
- 障害物の即時回避
- 経路の詳細な計算

### Q: Jetsonに渡されるデータ構成は？

サイクルごと（Mode A: 3秒 / Mode C: 5秒）に以下のJSON交換が行われる:
- **Jetson → Claude**: 状況JSON（全ロボットの位置、障害物、バッテリー、未完了タスク、履歴）
- **Claude → Jetson**: 指示JSON（reasoning + commands配列）

詳細フォーマットは `../architecture/08-llm-bridge-common.md` を参照。

### Q: LLMの見せ場は何か？

**Before/After比較**が最大の見せ場:
- Before（LLMなし）: 各ロボットが独立して最短経路 → 通路幅200mmですれ違えず渋滞・デッドロック
- After（LLM司令官）: 「Bot1は待機、Bot2は通路Bで迂回」→ スムーズに動く

加えて、Claude / ChatGPT / Gemini / Grok の判断の違いも比較できる。

### Q: ゲームマスター的な司令官のイメージか？

その通り。LLMは「誰が・どこへ・どの順番で」という高レベルな意思決定のみを行う。

```
LLM（司令官）     = 全体を俯瞰して戦略判断
Nav2（現場指揮官） = 具体的な経路計画・障害物回避
Bot1/Bot2（実行者）= 言われた通りにモーターを回す
```

## 2. 地図と座標 — 学習は発生するのか？

### Q: Nav2で地図を読み込む際に学習が発生するのか？

**学習は発生しない。RAGでもない。**

- minicar搭載のMS200 LiDARでジオラマ内をteleop走行し、SLAM Toolboxで地図（`map.pgm` + `map.yaml`）を生成する（1回だけ）。**Gazebo sim では SLAM を使わず `warehouse_sim.map_generator` が `layout` から決定論生成する**（同一形式。doc09 §7「地図の生成経路」, #76）
- Nav2はその地図ファイルを読み込んで経路計画に使う
- LLMには地図データは渡さない。場所名と座標の対応テーブル（`LOCATIONS`辞書）と現在の状況JSONを渡すだけ

### Q: 座標はピクセルデータか？距離の単位は？

**座標はメートル単位の実世界座標であり、ピクセルではない。**

`map.yaml` の `resolution` がピクセルとメートルの橋渡しをする:

```
実世界座標(m) = origin + ピクセル位置 × resolution
例: resolution=0.01 → 1ピクセル=1cm → ピクセル(30,50) = 実世界(0.30m, 0.50m)
```

Nav2の経路計算もすべてメートル単位で行われる。

## 3. AMCL — 自己位置推定の仕組み

### Q: パーティクルとは何か？

パーティクル = 「ロボットがここにいるかもしれない」という仮説の点。

地図上に数百〜数千個をばらまき、LiDARの実測データと各仮説位置の理論値を比較。一致しない仮説を消し、一致する仮説の周辺に新しい点を集中させることで、位置を1点に収束させる。

### Q: 処理速度は？一瞬で計算するのか？

一瞬ではなく、**1秒間に5〜10回のサイクルで繰り返す**（5-10Hz）。1回の計算は10〜30msで完了する。

| 処理 | 頻度 | 1回の計算時間 |
|------|------|-------------|
| AMCL（位置推定） | 5-10Hz | 10-30ms |
| グローバル経路計画 | ゴール変更時のみ | 50-200ms |
| ローカル制御（DWB/MPPI） | 20Hz | 1-20ms |
| LLM戦略判断 | 0.33Hz（3秒間隔） | 1,000-3,000ms |

## 4. minicarの位置特定 — 誰がどこで計算するか

### Q: minicarは自分の位置を知っているのか？

**minicar自身は自分の位置を知らない。** 位置を計算するのはJetson。

```
minicar（ESP32）の役割:
  - LiDARスキャンデータを送信
  - エンコーダの回転数からオドメトリ（移動量）を計算して送信
    ※「どれだけ動いたか」はminicar側で計算している
  - 受信した速度指令でモーターを回す
  - 「地図上のどこにいるか (x, y)」は計算しない

Jetsonの役割:
  - minicarからのセンサーデータ（scan, odom）を受信
  - AMCLで地図上の位置 (x, y, θ) を計算
  - Nav2で経路計画
  - cmd_velを計算してminicarに送信
```

minicarは「移動量」は知っているが「地図上の位置」は知らない。

人間の体に例えると:
- minicar = 目（LiDAR）と足（モーター）と三半規管（IMU）
- Jetson = 脳（位置計算、経路計画、判断）
- Claude = 司令官（戦略的判断）

### Q: minicarにはどのようなセンサーが搭載されているか？

Yahboom MicroROS ESP32 Car には以下のセンサーが搭載されている:

| センサー | 型番 | スペック | 用途 |
|---------|------|---------|------|
| dToF LiDAR | ORBBEC MS200 | 360°全方位, 0.03〜12m, 4500Hz, 角度分解能0.4° | AMCL自己位置推定・障害物検知 |
| 6軸IMU | 基板内蔵 | 加速度3軸 + ジャイロ3軸 | 姿勢推定・旋回検出 |
| エンコーダ | 310モーター内蔵 ×4 | 車輪回転数カウント | オドメトリ（移動量計算） |

**重要: ORBBEC MS200は360°スキャン可能なLiDARであり、単純な1方向ToFセンサーではない。** これにより標準的なAMCLがminicar単体で動作する。

### Q: 固定設置のRPLiDAR A1との関係は？

| センサー | 設置場所 | 役割 |
|---------|---------|------|
| ORBBEC MS200 | minicar搭載（移動する） | AMCL自己位置推定（常時）、障害物検知（常時） |
| RPLiDAR A1 | Jetsonに固定（俯瞰） | 外部トラッキング補正（オプションのみ） |

SLAM地図生成もAMCL自己位置推定も、minicar搭載のMS200を使用する。RPLiDAR A1は固定位置からでは棚裏の遮蔽があり完全な地図を作れないため、SLAMには使用しない。

### Q: 俯瞰センサーとminicarセンサーの融合で精度は上がるか？

**上がる。** これは「センサーフュージョン」と呼ばれる手法。各センサーの弱点を他のセンサーが補完する:

- **固定LiDARの弱点**: 棚の裏が見えない（遮蔽） → オドメトリが補完
- **オドメトリの弱点**: 累積誤差が大きくなる → 固定LiDARが補正
- **IMUの弱点**: 長時間でドリフトする → LiDARとオドメトリが補正

実装順序:
1. Phase 2a: minicar ORBBEC MS200 + AMCL で動作確認（最小構成）
2. Phase 2b: 固定RPLiDAR A1の外部トラッキングを追加（精度向上オプション）
3. Phase 2c: IMU融合を追加（余裕があれば）

## 5. 処理ループ — 常時動作かイベント駆動か

### Q: 位置推定や経路計画はminicar走行中も常に動いているのか？

**常に動き続けている。** 「何かのアクションで1回実行」ではなく、複数のループが異なる頻度で同時並行に回り続ける。

### 常時回り続けるもの（停止中も動作）

| 処理 | 頻度 | 停止中の動作 |
|------|------|------------|
| MS200 LiDAR スキャン | 4500Hz（送信は10-15Hz） | 回転し続けてデータを送信 |
| エンコーダ報告 | 20-50Hz | 「回転0」を送信し続ける |
| AMCL 位置推定 | 5-10Hz | 「位置は変わっていない」と確認し続ける |
| DWB/MPPI 速度計算 | 20Hz | 「速度0を維持」と計算し続ける |

常に監視する理由: ロボットが手で押された場合や床で滑った場合にも位置を見失わないため。

### 定期実行（タイマー駆動）

| 処理 | 頻度 | 内容 |
|------|------|------|
| LLM Bridge | Mode A: 3秒 / Mode C: 5秒サイクル | 状況JSONをClaudeに送信し指示を受け取る |

### イベント駆動（何かが起きたときだけ動作）

| 処理 | トリガー |
|------|---------|
| Nav2 グローバル経路計画 | 新しいゴールが来たとき |
| リカバリー（回転・後退） | ロボットが詰まったとき |

### 1秒間の時間軸

```
0ms     100ms   200ms   300ms   ...   900ms   1000ms
│       │       │       │             │       │
│ scan ─┤─ scan ┤─ scan ┤─ scan ...   ┤─ scan ┤  10-15Hz
│ odom ──┼──┼──┼──┼──┼──┼──┼──┼──...──┼──┼──┼─  20-50Hz
│ AMCL ─┤       ├─ AMCL ┤       ...   ├─ AMCL ┤  5-10Hz
│ MPPI ─┼──MPPI─┼──MPPI─┼──MPPI─...──┼──MPPI─┼  20Hz
│                                               │
│ LLM ─────── Mode A:3秒 / Mode C:5秒 ────────  │  0.33-0.2Hz
```

**LLMだけが3秒間隔のタイマー駆動**であり、それ以外は全て常時回り続ける。これにより、LLMの応答が遅れてもAMCLとNav2が常に安全にロボットを制御し続けることができる。

## 6. コストマップと経路計算

### Q: コストマップは距離計算にも使われるか？

**Nav2の経路計算はすべてコストマップ上で行う。** 単純なユークリッド距離ではなく、壁の近さを考慮した「最も安全で効率的な経路」を選択する。

```
経路コスト = Σ（セル間の移動コスト + そのセルのコスト値）
```

壁に近いセルはコスト値が高いため、距離が多少長くても壁から離れた経路が選ばれる。

### Q: DWBは速度も変数に入れて計算しているのか？

**はい。** DWBは各サイクル（20Hz）で以下を実行:
1. 現在速度(v, ω)から物理的に到達可能な速度候補を複数生成
2. 各速度で数秒先まで走行した軌道をシミュレーション
3. コスト評価（障害物、経路逸脱、目標距離、速度効率）で最適な速度を選択
4. `/cmd_vel` として出力

速度も経路計画の重要な変数であり、「どの速度で走れば安全かつ効率的か」を毎サイクル計算している。

## 7. 自動運転技術との比較

### Q: テスラもこのような処理を高速に行っているのか？

**基本構造（認識→位置推定→経路計画→制御）は同じ。** ただしスケールが桁違い。

処理速度・相違点の詳細テーブルは `09-navigation-internals.md` セクション6を参照。

要点のみ記載:
- テスラFSDのエンドツーエンドは推定50-100ms、本プロジェクトは約3秒（LLM含む）
- テスラは3D+時間軸の4D予測、本プロジェクトは2D静的環境
- テスラは全処理を車載チップで完結（クラウド通信なし）、本プロジェクトはLLMのみクラウド

※テスラFSDの内部処理速度は公式に公開されていない。推定値は公開情報・業界推定に基づく参考値。

### 本プロジェクト独自の要素

テスラにない要素として:
- **複数台の協調制御**: LLMが全体を俯瞰して交通整理（テスラは各車が独立判断）
- **判断理由の可視化**: reasoningが人間に読める形で出力
- **LLM比較検証**: 同じ状況でClaude/ChatGPT/Gemini/Grokの判断の違いを定量比較

## 8. JetsonとClaudeの責任分担

### Q: 常時の衝突回避と戦略判断はどう分けるべきか？

**50ms以内に判断が必要 → Jetson、数秒の猶予 → Claude。**

Jetson（Nav2 + Emergency Guardian）の担当:
- 壁・棚・ロボット同士の物理的衝突回避（Nav2 DWB/MPPI、50ms）
- ゴールまでの経路追従・速度調整（Nav2 Controller、50ms）
- 一時的なスタックからのリカバリー（Nav2 Recovery、3回失敗でOpen-RMFにエスカレーション）
- 速度上限の強制（ESP32/MCU内で0.3m/sにクランプ + Nav2パラメータ）
- 2台間距離監視・バッテリー監視・blocked検出（Emergency Guardian、50ms周期、LLM非経由）

Claude（LLM）の担当（モードC: Open-RMF主方針）:
- タスク割当（誰がどの仕事をやるか）
- タスク優先順位変更
- バッテリー管理
- エスカレーション対応（Open-RMFが解決できなかった問題のみ）

※デッドロック解消・渋滞の予防・迂回ルート指示はOpen-RMFが自動処理する。

詳細は `../architecture/08-llm-bridge-common.md` の「3層責任分担」セクションおよび `../architecture/12-infrastructure-common.md` の安全レイヤー設計を参照。

### Q: LLMの3秒間隔で齟齬は起きないか？

**起きうる。** 4層の安全レイヤーで対策する（`../architecture/12-infrastructure-common.md` 参照）:

- Layer 0（ESP32/MCU、即時）: ハードウェア安全停止（最終防衛線、通信不要）
- Layer 1（Emergency Guardian、50ms周期）: ソフトウェア安全監視（距離・バッテリー・blocked検出）、LLM非経由
- Layer 2（Open-RMF、即時）: 交通管理（経路衝突予測・待機・迂回指示）
- Layer 3（Claude、3秒）: 戦略判断（タスク再割当・復旧方針の提案）

## 9. デッドロックとMulti-Robot Costmap Layer

### Q: デッドロックとは何か？

2台以上が互いに「相手が動くのを待っている」状態で、誰も動けなくなること。通路幅200mm（すれ違い不可）の本プロジェクトでは高頻度で発生しうる。

- 検出: LLM Bridge（Jetson側）で「両方blocked かつ互いに近い」を判定
- 解消: Claude（LLM）が「誰がどう退くか」を判断

### Q: Nav2は他のロボットを自動で認識できるか？

**LiDARの視野内なら検出できるが、死角は検出できない。** 確実に認識するには、各ロボットのAMCL出力を相手のコストマップに動的障害物として注入するMulti-Robot Costmap Layer（自作）が必要。

実装Phase:
1. Phase 2a: LiDARの直接検出のみ（正面は認識可能）
2. Phase 2b: Multi-Robot Layerを自作して追加（死角を解消）

### Q: 他のロボットの速度を予測して避けることはできるか？

**できる。** MPPIコントローラーは「現在位置+速度→未来位置」を予測してコスト計算に含める。Multi-Robot Layerを拡張し、相手の速度情報も注入すれば予測ベースの回避が可能。ただしPhase 2では0.3m/s・2台なのでDWB/MPPIの50ms反応で十分。

## 10. DWBとMPPIコントローラー

### Q: MPPIとは何か？

MPPI = Model Predictive Path Integral（エム・ピー・ピー・アイ）。Nav2に標準搭載されたローカルコントローラー。追加費用・学習不要。yaml 1行で切り替え可能。

DWBとの違い:
- DWB: 格子状に約30個の速度候補を生成、最小コストの1つを選択
- MPPI: ランダムに1,000〜2,000本の軌道を生成、コストで重み付けした加重平均で最適速度を算出

### Q: MPPIの計算負荷は問題にならないか？

Jetson Orin Nano Super（67 TOPS、6コア）では問題にならない。MPPI×2台でもCPU使用率40-60%程度。20Hzを維持可能（最悪15Hz、0.3m/sでは十分安全）。

### 方針

Phase 2前半: DWBで動作確認 → Phase 2後半: MPPIに切り替え → Phase 3以降: MPPI前提でLLM統合

MPPIを推奨する理由: 狭い通路（200mm）での正確な走行、滑らかな動き（YouTube映像品質）、Jetsonで計算負荷が問題にならない。

## 11. predicted_position_3s — LLMの判断遅延対策

### Q: predicted_position_3sとは何か？何のためにあるか？

Claudeが3秒間の判断遅延を見越した戦略判断を行うための補助データ。現在の速度・方向で3秒間まっすぐ進んだ場合の位置を線形外挿で計算する。

### Q: 具体的にどの場面で必要か？

**渋滞の予防**が最大の目的:

- positionだけの場合: 「2台は0.6m離れている、問題なし」→ 3秒後にデッドロック
- predicted_position_3sがある場合: 「3秒後に同じ通路で鉢合わせする」→ 今のうちに迂回指示

その他の場面:
- タスク割当: 3秒後の位置で「どのBotがどのタスクに近いか」を判断
- 充電判断: 充電ステーションから離れている方向に進んでいるなら早めに戻す

### Q: なぜ正確な予測ではなく線形外挿なのか？

正確な予測（Nav2の計画経路上の3秒後位置）は以下の理由で採用しない:
- 経路自体がDWB/MPPIによって50msごとに変わる
- 計算が複雑になる割に、Claudeの戦略判断には精密な位置は不要
- 精密な衝突回避はNav2（50ms）が担当するため、Claudeには方向性の情報で十分

限界: 曲がり角・壁・ゴール到着を考慮しないため、物理的にありえない位置を予測することがある。Claudeには「2台が近づいている/離れている」という方向性の参考値として提供する。

詳細は `../architecture/08-llm-bridge-common.md` の「predicted_position_3s について」セクションを参照。

## 12. 交通管理レイヤーとOpen-RMF

### Q: Open-RMFとは何か？

Open Robotics（ROS 2の開発元）が開発したマルチロボットフリート管理フレームワーク。ROS 2上で動作し、Traffic Schedule（経路予測・衝突検出）とConflict Negotiation（衝突時の自動調整）を提供する。Apache 2.0ライセンスで無料、Jetsonで動作可能。

### Q: なぜプラグイン方式で交通管理を設計するのか？

**主方針はモードC（Open-RMF）。** 交通管理はOpen-RMFが即時処理し、Claudeはタスク割当・優先順位・バッテリー管理の戦略判断のみを行う。モードA/Bはconfig.yaml切替でYouTube比較検証用に使用する。

| モード | 交通管理 | Claudeの責任 | YouTube映像での見え方 |
|--------|---------|-------------|---------------------|
| A: none | Claudeが全判断 | 交通管理含む全判断 | Claudeの判断力が最も見える |
| B: simple | 通路ロック（自作） | 戦略 + 一部交通 | 即時の排他制御あり |
| **C: open-rmf（主方針）** | **Open-RMFが即時処理** | **タスク割当・優先順位・バッテリーのみ** | **交通管理の高度さが見える** |

config.yamlの1行変更で切り替え可能。

### Q: ClaudeとOpen-RMFはどう通信するのか？

LLM Bridgeが仲介する。Claudeの指示はTrafficManagerを経由してNav2に送られ、TrafficManagerの交通状態はClaudeの状況JSONのtrafficセクションに含められる。

通信ルール:
- Claudeは「何をするか」を決め、TrafficManagerは「どう実現するか」を決める
- TrafficManagerの調整中はClaudeは介入しない
- 3回調整失敗でClaudeにエスカレーション
- Claudeが強制的にTrafficManagerを無視する仕組みは設けない（安全のため）

### Q: predicted_position_3sはOpen-RMF導入後どうなるか？

モードC（Open-RMF）では経路ベースの正確な予測が得られるため、predicted_position_3s（線形外挿）は不要になる。モードA/Bでは引き続き使用する。

詳細は `../mode-a/11a-traffic-mode-a.md` および `../mode-c/11c-traffic-mode-c.md` を参照。

## 13. ROS 2の守備範囲

### Q: ROS 2は3Dや音声の処理も行うのか？

ROS 2自体は「ノード間の通信基盤」であり、処理は各ノードが行う。本プロジェクトでは2D処理のみ使用。

| 領域 | ROS 2で可能か | 本プロジェクトで使用 |
|------|-------------|-------------------|
| 2D地図・経路計画 | Yes | **Yes**（SLAM + Nav2） |
| 3D地図（OctoMap） | Yes | No |
| 3D認識（FoundationPose） | Yes | Phase 5+で検討 |
| 深度カメラ | Yes | No |
| 音声認識・合成 | Yes | No |
