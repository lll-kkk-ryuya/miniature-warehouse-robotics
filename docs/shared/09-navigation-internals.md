# ナビゲーション内部処理の詳細

作成日: 2026-05-21
更新日: 2026-05-22

## 概要

本ドキュメントでは、AMCL（自己位置推定）、Nav2（経路計画・走行制御）、コストマップの内部処理を解説する。自動運転技術との共通点・相違点についても整理する。

---

## 1. 地図ファイルの構成

SLAM Toolboxでスキャンすると2つのファイルが生成される。

### map.pgm（画像ファイル）

- グレースケール画像（白=通行可、黒=壁/障害物、灰色=未探索）
- 各ピクセルが地図上の1セルに対応

### map.yaml（メタデータ）

```yaml
image: map.pgm
resolution: 0.01         # 1ピクセル = 0.01m（1cm）
origin: [0.0, 0.0, 0.0]  # 地図の左下隅の実世界座標 (x, y, yaw)
occupied_thresh: 0.65
free_thresh: 0.196
negate: 0
```

### ピクセル-実世界座標の変換

```
実世界座標(m) = origin + ピクセル位置 × resolution
```

- 例: `resolution: 0.01` の場合、ピクセル(30, 50) → 実世界(0.30m, 0.50m)
- 1.8m×0.9mのジオラマ → 180×90ピクセルの画像

座標はメートル単位の実世界座標であり、ピクセル座標ではない。`resolution`がピクセルとメートルの橋渡しをする。

## 2. AMCL（自己位置推定）

### 概要

AMCL = Adaptive Monte Carlo Localization（適応的モンテカルロ位置推定）。パーティクルフィルタという確率的手法で「自分が地図上のどこにいるか」を推定する。

### パーティクルとは

パーティクル = 「ロボットがここにいるかもしれない」という仮説の点。地図上に数百〜数千個をばらまき、センサーデータとの整合性が低い仮説を消し、高い仮説の周辺に新たな点を集中させることで位置を収束させる。

### 処理ステップ

```
Step 1: パーティクル散布
  地図上に500〜2000個の仮説点を散布

Step 2: センサー照合
  LiDAR実測値と、各パーティクル位置から計算した理論値を比較
  → 一致度が高いパーティクルの確率を上げる

Step 3: リサンプリング
  確率の高いパーティクル周辺に新しいパーティクルを集中
  → 繰り返すと1点に収束 = 自己位置確定

Step 4: 出力
  /bot{n}/amcl_pose → (x, y, θ) メートル・ラジアン単位
```

### 入出力

| 入力 | ソース | 内容 |
|------|--------|------|
| 地図 | `map.pgm` + `map.yaml` | 壁の配置 |
| LiDARデータ | ORBBEC MS200（minicar搭載, 360°）→ `/bot{n}/scan` | 周囲の壁までの実測距離 |
| オドメトリ | ESP32エンコーダ → `/bot{n}/odom` | 車輪回転数からの移動量 |
| IMU | 6軸IMU（minicar搭載）→ `/bot{n}/imu`（※要確認） | 加速度・角速度（姿勢補助） |

**minicar搭載のORBBEC MS200（360° dToF LiDAR）により、標準的なAMCLがそのまま動作する。** ロボットが移動するとLiDARの見え方が変わるため、AMCLの前提条件を満たす。

### センサーの役割分担

| センサー | 設置場所 | 役割 |
|---------|---------|------|
| ORBBEC MS200 | minicar搭載（移動） | AMCL自己位置推定（常時）、障害物検知（常時） |
| RPLiDAR A1 | Jetsonに固定（俯瞰） | 外部トラッキング補正（オプション） |
| エンコーダ | minicar搭載 | オドメトリ（移動量、常時） |
| 6軸IMU | minicar搭載 | 姿勢推定・旋回検出（補助） |

### センサーフュージョン（精度向上オプション）

minicar単体（MS200 + エンコーダ + IMU）でAMCLは動作するが、固定RPLiDAR A1の外部トラッキングを融合すればさらに精度を向上できる。

```
minicarセンサー（自分視点）:
  MS200 LiDAR → AMCL → 位置推定        信頼度: 高
  エンコーダ  → オドメトリ → 移動量     信頼度: 中
  IMU        → 姿勢推定               信頼度: 中

固定センサー（俯瞰）:
  RPLiDAR A1 → 外部トラッキング → 補正  信頼度: 高（遮蔽がなければ）

融合:
  minicar AMCL + 固定LiDAR補正 → 高精度な位置推定
  ※固定LiDARによる補正はオプション。minicar単体でも動作可能。
```

### SLAM地図生成に使用するセンサー

SLAM Toolboxはセンサーを移動させながらスキャンを蓄積して地図を構築する。固定設置のRPLiDAR A1では1地点からのスキャンしか得られず、棚の裏側など遮蔽領域が地図に反映されない。

**SLAM地図生成にはminicar搭載のMS200を使用する。** minicarをteleop（手動操縦）でジオラマ内を走り回らせ、MS200の360°スキャンデータをSLAM Toolboxに入力して完全な地図を構築する。RPLiDAR A1はSLAMには使用しない。

実装順序:
1. Phase 2a: minicar ORBBEC MS200でSLAM地図生成 + AMCL動作確認（最小構成）
2. Phase 2b: 固定RPLiDAR A1の外部トラッキングを追加（精度向上オプション）
3. Phase 2c: IMU融合を追加（余裕があれば）

| 出力 | トピック | 内容 |
|------|---------|------|
| 推定位置 | `/bot{n}/amcl_pose` | (x, y, θ) |
| TF変換 | `map → odom` | 座標系の補正 |

## 3. Nav2（経路計画・走行制御）

### コストマップ

Nav2は`map.pgm`をそのまま使わず、コストマップに変換する。壁の周囲にコスト（危険度）のグラデーションを付けることで、ロボットが壁に近づきすぎないようにする。

```
コスト値:
  0     = FREE_SPACE（安全、壁から十分離れている）
  1-252 = 壁に近いほどコストが高い（inflation layer が生成）
  253   = INSCRIBED（ロボットのfootprint内に壁がある、衝突確定）
  254   = LETHAL（壁そのもの、通行不可）
  255   = NO_INFORMATION（未探索領域、通行可否不明）
```

### 経路計算（Dijkstra / A*）

コストマップをグリッドとして扱い、各セル間の移動コストを計算する。

```
経路コスト = Σ（セル間の移動コスト + そのセルのコスト値）

セル間の移動コスト:
  直進（上下左右）:  resolution × 1 = 0.01m相当
  斜め:             resolution × √2 ≈ 0.0141m相当

各セルのコスト値（0〜254）が加算される
→ 壁に近いセルを通ると経路コストが大きくなる
```

Nav2は「最短距離」ではなく「最も安全で効率的な経路」を選択する。

### ローカルコントローラーと速度変数

#### DWB（Phase 2前半で使用）

DWB（Dynamic Window Approach）は、格子状に速度候補（約30個）を生成し、コストマップ上でシミュレーションして最小コストの速度を選択する。設定が簡単で計算が軽い。

#### MPPI（Phase 2後半以降で使用、推奨）

MPPI（Model Predictive Path Integral）は、ランダムにノイズを加えた軌道候補を大量（1,000〜2,000本）生成し、各軌道のコストに基づく重み付き加重平均で最適速度を算出する。Nav2 Jazzyに標準搭載されており、yaml 1行の変更で切り替え可能。追加コスト・学習は不要。

| 特徴 | DWB | MPPI |
|------|-----|------|
| 候補数 | 約30個（格子状） | 1,000〜2,000本（ランダム） |
| 選択方法 | 最小コストの1つを選択 | 全候補の重み付き平均 |
| 計算時間 | 1-5ms | 5-20ms |
| 動きの滑らかさ | 角張ることがある | 滑らか |
| 狭い通路 | 候補が少なく苦手 | 大量候補で得意 |
| Jetson負荷 | CPU 10-20% | CPU 20-50%（余裕あり） |

MPPIを推奨する理由:
- 通路幅200mmの狭い環境でDWBより正確な走行が可能
- 動きが滑らか（YouTube映像品質の向上）
- Jetson Orin Nano Super（67 TOPS）で計算負荷は問題にならない
- 20Hzを維持可能（最悪15Hz、0.3m/sでは十分安全）

移行計画:
1. Phase 2前半: DWBで動作確認（設定が簡単、まず動かす）
2. Phase 2後半: MPPIに切り替え（デフォルトパラメータで試行）
3. Phase 3以降: MPPI前提でLLM統合

```
各サイクル（50ms = 20Hz）で以下を実行:

1. 速度候補の生成
   現在速度(v, ω)から加速度制約の範囲内で到達可能な
   速度の組み合わせを複数生成
   例: (0.1m/s, 0.0rad/s), (0.15m/s, 0.3rad/s), ...

2. 軌道シミュレーション
   各速度候補で数秒先まで走行した場合の軌道を予測
   コストマップ上で衝突チェック

3. コスト評価
   - 障害物への近さ（コストマップのコスト値）
   - 目標経路からの逸脱度
   - 目標地点への距離
   - 速度（速いほうが効率的）
   → これらの重み付き合計で最適な速度を選択

4. 出力
   /bot{n}/cmd_vel → (linear.x, angular.z)
```

速度も経路計画の重要な変数であり、「どの速度で走れば安全かつ効率的か」を毎サイクル計算している。

## 4. 処理速度

| 処理 | 頻度 | 1回の計算時間 |
|------|------|-------------|
| AMCL（位置推定） | 5-10Hz | 10-30ms |
| グローバル経路計画 | ゴール変更時のみ | 50-200ms |
| ローカル制御（DWB: 1-5ms / MPPI: 5-20ms） | 20Hz | 1-20ms |
| cmd_vel出力 | 20Hz | — |
| LLM戦略判断 | 0.33Hz（3秒間隔） | 1,000-3,000ms |

## 5. Jetson上で動くROS 2ノード一覧

```
┌─────────────────────────────────────────────────┐
│  micro-ROS Agent       ESP32とのWiFi通信中継     │
│  RPLiDAR Driver        LiDARスキャンデータ取得    │
│  Map Server            map.pgmを読み込んで配信    │
│  AMCL × 2             自己位置推定（Bot1/Bot2各1） │
│  Nav2 Planner × 2     グローバル経路計画（各Bot）  │
│  Nav2 Controller × 2  ローカル制御・DWB→MPPI（各Bot）│
│  Nav2 Recoveries × 2  リカバリー・回転・後退      │
│  Costmap Node × 2     コストマップ生成・更新       │
│  Lifecycle Manager     各ノードの起動・停止管理    │
│  Emergency Guardian    安全監視（50ms周期、LLM非経由）│
│  State Cache Node      状態集約JSON書出し（100ms周期）│
│  LLM Bridge Node       Hermes Gateway連携（Mode A:3秒/Mode C:5秒）│
│  Hermes Gateway        LLM推論daemon（port 8642）   │
│  Warehouse MCP Server  Policy Gate + Nav2/RMF制御   │
│  WO Bridge Node        Webダッシュボード連携       │
│  TF2                   座標変換管理               │
└─────────────────────────────────────────────────┘
```

### Map Server の共有方式

1つのMap Serverが `/map`（グローバルnamespace）を配信し、Bot1/Bot2両方のAMCLインスタンスが購読する。同じ地図を使うため、Map Serverは1つで十分。

### TFツリー構造（2台構成）

```
map（グローバル、共有）
├── bot1/odom（AMCL_1が管理、map→odomの変換を配信）
│   └── bot1/base_link
│       ├── bot1/lidar_link（MS200）
│       └── bot1/imu_link
└── bot2/odom（AMCL_2が管理、map→odomの変換を配信）
    └── bot2/base_link
        ├── bot2/lidar_link（MS200）
        └── bot2/imu_link
```

各ロボットはROS 2のnamespace（`/bot1/`, `/bot2/`）で分離する。`map`フレームのみグローバルで共有。
```

## 6. 自動運転技術との比較

### 共通する処理パイプライン

自律移動ロボットも自動運転車も「認識→位置推定→経路計画→制御」の4段階は同じ。

### 処理速度の比較

※テスラFSDの内部処理速度は公式に公開されていない。以下の数値は公開情報・業界推定に基づく参考値。

| 項目 | 本プロジェクト | 自動運転車（テスラFSD等、推定値） |
|------|--------------|-------------------------------|
| 認識サイクル | 5-10Hz | 30-60Hz |
| 位置推定 | 10-30ms | <10ms |
| 経路計画 | 50-200ms | <50ms |
| 障害物回避 | 50ms | <20ms |
| 戦略判断 | 1,000-3,000ms（クラウドLLM） | <50ms（車載NN） |
| エンドツーエンド | 約3秒 | 約50-100ms |

### 主要な相違点

| 観点 | 本プロジェクト | 自動運転車 |
|------|--------------|-----------|
| 次元 | 2D（平面走行） | 3D + 時間軸（4D予測） |
| 環境 | 静的（壁・棚は動かない） | 動的（車・歩行者が動く） |
| センサー | 2D LiDAR 3台（minicar搭載ORBBEC MS200×2 + 固定RPLiDAR A1×1） | カメラ8台+各種センサー |
| 地図 | 事前スキャン（静的） | リアルタイム3D再構築 |
| 判断 | クラウドLLM（遅延許容） | 車載チップ（遅延不可） |
| 速度 | 最大0.3m/s | 最大30m/s以上 |
| 失敗時 | ロボットが停止する | 人命に関わる |
| 協調 | LLMが複数台を中央集権的に制御 | 各車が独立判断 |

### 本プロジェクト独自の要素

- **複数台の協調制御**: 自動運転車は各車が独立判断するが、本プロジェクトではLLMが全体を俯瞰して交通整理を行う
- **LLMによる自然言語での戦略判断**: 判断理由（reasoning）が人間に読める形で出力される
- **LLM比較検証**: 同じ状況でClaude/ChatGPT/Gemini/Grokの判断の違いを定量比較できる
