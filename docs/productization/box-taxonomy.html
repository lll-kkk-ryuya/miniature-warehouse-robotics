Productization Box 切り分け図解

L4 Robotics Bridge Super-Box を中心とした「再利用 box / sub-box / seam / plugin / demoted」の分類と、各要素が紐づく設計 docs。

正本: docs/productization/01-commercial-box-map.md §Box 種別と分類規則 ／ 本ページはその図解（補足資料）。ブランチ docs/productization-box-taxonomy。 種別の定義・file:line は必ず 01〜08 の .md を一次ソースとする。

あなたの「box とは個々を組み合わせて再利用できる単位」という原則に対し、元の docs は3つの食い違う分類（map=Hermes を1箱に隔離 / 表=各箱を Hermes列・Bridge列に縦割り / 小設計=4箱）が混在し、seam（境界の写像）まで「box」として並んでいました。これを 1つの種別体系に統一したのが今回の切り分けです。

0. 切り分けの 5 種別（凡例）

box安定した produces/consumes 契約を持ち、契約を変えず丸ごと差し替え/省略できる再利用単位。実体は ROS package と1:1とは限らない。
sub-box親 box の入出力の一部を担う内部ステージ。親 interface を越えて単独 consume されない。親ごと差し替え。
seam2 box 境界の写像/変換。新 schema を産まず出力が隣接 box の入力契約そのもの。常時随伴で省略不可。所有 box 配下。
pluginbox の interface を変えず中身の実装差を吸収する差替軸。box でなく box 内部の注入点。
demotedbox でないもの（Hermes transport 実装 / 設計実体ゼロ / seam の box 誤列挙）。所有 box の責務へ吸収。
核心原則 — transport は box を割らない: 「Hermes 経由か direct か」は box の interface 裏の実装選択 （transport: hermes | direct | worker）であって、箱の真ん中を Hermes/Bridge 線で割る話ではない。 だから旧 docs の「縦割り表」は box 境界ではなく transport 実装ノートとして読み替えた。

4. なぜ「縦割り」をやめたか（box を割らない）

❌ 旧 docs: 箱を Hermes/Bridge で縦割り

Hermes 列：provider 切替・fallback …
Bridge 列：request id・0 dispatch …

1つの「Model Transport Box」の真ん中を境界線が貫く。これでは 箱として入れ替えられない（半分 Hermes・半分 Bridge）。

✅ 新: 1 box ＋ transport は内部の実装選択

Model Adapter sub-box
contract（produces/consumes）は1つ。差し替え単位。
transport: hermes | direct | worker（interface 裏）

box は1ユニットのまま。Hermes 採用は内部の実装選択なので、Hermes を direct/worker に丸ごと差し替え可能。これが再利用の土台。

安全核は box の真ん中を貫けない: MAX_LINEAR_VELOCITY=0.3（warehouse_interfaces/safety.py:18）/ gen_id・idempotency_key mint（action_map.py＝Bridge mint・LLM 由来不可）/ KNOWN_LOCATIONS 9キー（locations.py）/ 0 dispatch は上位 model 非依存。 だから Governance / Safety / Hardware の transport は n/a（hermes と書くと motion gate が Hermes 線で貫かれる）。

6. まだ綺麗に割り切れていない点（要判断）

機械チェック（check_consistency 0 ERROR）＋意味監査（/consistency-audit）は通し、実エラー2件（safety.py のパス・manifest status 自己矛盾）は修正済み。その上で残る判断点:

  • ① L3 Core の optional が経路依存: 現状 Mode A/B/C は LLM の Command JSON → action_map 直行で L3 Core を経由しない。L3 Core は Mode X-ER で初めて実体化＝「Mode によって box 境界が動く」。図中 ※
  • ② Basic Vision が3分割: transport:hermes（画像入力）＋ Input Context（camera/calibration）＋ L3 Visual Resolver（map 解決）。provider_type enum に vision が無いため発明せず分配した結果＝唯一「1 box 1 interface」がフレる箇所。
  • ③ Provider Routing=plugin vs Model Transport=demoted: どちらも Hermes transport。routing を差替軸=plugin、transport 本体を demoted と分けたが境界は柔らかい。
  • ④ Fusion の置き場: Super-Box 配下 sub-box としたが 02:188「VLA は必ず Fusion→L3 を通す」で L3 と境界が重なる。L3 内 Validator 内包案も残る。
  • ⑤ #319 とのレイヤ軸衝突: 別 PR #319 が同じ 01 に L4–L0 レイヤ列を追加中。レイヤ⊥種別で直交＝最終形は両方併記が理想（merge 順は要調整）。

①〜④は 01〜03 の Deferred open-decisions に列挙済み（contract 昇格・Mode 依存の表現は別 contract PR 案件）。
