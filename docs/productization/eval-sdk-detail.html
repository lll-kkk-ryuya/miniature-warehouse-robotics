eval-sdk 詳細図 — 横断 観測・評価 基盤（Observability & Eval）

eval_sdk は L3 のような縦の box ではなく、L4〜L0 を横断する observe-only（観測専用）の再利用基盤。producer（各層）→ eval_sdk（汎用 5 モジュール）→ Langfuse（保管・dashboard）の 3 層で、判断と結果を 1 本の trace に結合する。パッケージは実装済（ws/src/eval_sdk/・doc21 Phase 1 抽出済・倉庫が一方向依存で配線済）／API は未凍結契約（Phase 2 で ScoreSpec 名のみ凍結）。enforcement には絶対に触れない・fail-open。正本 architecture/21。

0. 位置づけ — box ではなく「横断 observe-only サイドカー」

L3/L4 のような縦のパイプライン box（入力→処理→出力で実行に効く）とは違い、eval_sdk は全層の decision_event / trace を読んで集計・保管するだけの横断層。所有は観測層、各層に producer が散らばる。故障しても走行は無傷（fail-open）。21:43-119,138

① producer（ドメイン＝倉庫固有・差し替える）
② eval_sdk（汎用＝一度作れば再利用・★これを作る）
③ Langfuse（既製＝乗るだけ）

1. 3 層アーキテクチャ（producer → eval_sdk → Langfuse）21:43-119

① producer（ドメイン＝倉庫・各 sim/研究で差し替える）差替
生信号を出すだけ（判断 / pose / event / tokens）。LLM Bridge の judgment、Nav2 の odom、MCP audit、Emergency Guardian の near_collision（観測タップ）、deadlock detector など。「何が起きたか」を出す。21:82-86,104,113
▼ 生信号（raw events / pose / tokens）
② eval_sdk（汎用・産業非依存・★作る）再利用
生信号を指標へ変換・join-key 付与・emit。seed が判断と結果を同じ trace_id に結合、stats が指標化（SR/SPL/介入率/percentile…）、sink が fail-open 送信。倉庫を知らない＝製造でもそのまま再利用。21:89-95,114,117
▼ trace + score（すべて同じ trace_id でひも付く）
③ Langfuse（既製 SaaS＝乗るだけ）保管
保管 + 可視化 + 比較。1 trace = 【判断】generation(input/output/latency/cost) + 【結果】score(SR/SPL/efficiency/deadlock/collision_free…)。4 provider × 3 mode 比較ダッシュボード / Metrics API / 可搬 export。21:98-101,115
層役割所有 / 再利用性各プロジェクトで
① producer生信号を出す（判断・pose・event・tokens）ドメイン（倉庫固有）差し替える（sim/研究ごとに別 producer）
② eval_sdk生信号 → 指標へ変換・join-key 付与・emit汎用（★これを作る）そのまま再利用（接続するだけ）
③ Langfuse保管・可視化・比較既製 SaaS乗るだけ（差し替え不要）

三者の役割・所有: 21:109-117

2. v0.1 = 5 モジュール（既存コードの抽出＝リネーム＋重複解消・抽出済 ws/src/eval_sdk/）21:123-135

巨大プラットフォームでなく、既にテスト済・脱ROS・fail-open のコードを near-verbatim で lift した 5 つの薄いモジュール。発明ではない。doc21 Phase 1 で抽出済（seed/tracer/sink/stats/cost.py・PR #273・倉庫が from eval_sdk import … で配線済）。21:15,125,237

核心 = deterministic join key: derive_trace_id(seed="run:work")。あらゆる emitter（判断・結果・sim・scorer・Langfuse sink）が同じデータから同じ trace_id を再導出する → 「判断 ↔ 動作」が同じ 1 本の trace に結合する。これが「リコール調査は 1 クエリ」「provider 比較」を成立させる土台で、L3 ③ audit 1:1 を横断で実現する機械側。2 階層のまま（emitter が出さない episode 階層に拡張しない）。21:74（derive_trace_id 署名・2階層）,90,117,143 / 03:235（L3 側の要件）

3. 背骨（不変条件・verbatim 維持）21:137-143

fail-open
creds 無・SDK 未導入・通信障害は静かに no-op（raise しない）。観測が落ちても走行は無傷。21:138
lazy-import
langfuse は任意 extra。未導入でも build/test 可（ハード依存を入れない）。21:139
依存注入
create_fn（trace_id 生成）・価格表を引数注入 → SDK 無しで単体テスト可能。21:140
死守する 1 不変条件: 「プロセス A で導出した trace_id == プロセス B で同 seed から導出した trace_id」の property test（join バグ #115 の修正を一般化・assert する）。これが全設計の土台。21:142-143

4. L4〜L0 のどこと関わるか（producer マップ）21:104-119 / 05:75-105

5. enforcement 非接触の境界（最重要・安全）21:119

eval_sdk は enforcement 経路に絶対に触れない。Emergency Guardian / collision_monitor / 0.3 m/s クランプは fail-closed の能動制御であって ① の「観測 producer」ではない。eval が読むのはその観測専用の複製イベント（near_collision の count 等）だけで、enforcement 経路には触れない。producer 図中の Guardian は「観測タップ」を指す。eval_sdk の故障（creds 無・通信断）は走行を一切止めない（fail-open・§3）。21:119,138
兄弟関係: reject 理由を人へ音声で返す L4 Operator Feedback Box は、eval_sdk と同じ横断 decision_event バスを読む兄弟。違いは責務 — eval_sdk は observe-only の集計/KPI/report sink、Operator Feedback は リアルタイムに人へ出す一次チャネル。どちらも実行許可を持たず fail-open。05 §observe / mode-x-er/05 §2.1
