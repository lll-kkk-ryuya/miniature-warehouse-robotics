0. なぜ分けるのか — Langfuse は live feed の代替に物理的になり得ない

低い layer ほど呼び出し頻度が上がる（Safety は 50ms tick、Hardware は cmd_vel/heartbeat/sensor）。これを丸ごと Langfuse trace に入れると量が爆発し、肝心の「判断」が埋もれる。05:109,122

結論（doc22:132 明言）: 「ライブの真実 = ROS via web_bridge、耐久・比較の真実 = Langfuse + audit.jsonl、両者は trace_id で join。Langfuse はライブ feed の代替に物理的になり得ない」。よって 高頻度 ROS signal を常時 Langfuse へ投げることは Eval box の禁止事項（07:59）、Langfuse を高頻度 safety enforcement path にすることは Safety box の禁止事項（07:57）。22:132 / 07:57,59

event tier（audit.jsonl・event 時）
集計 tier（metrics・常時だが潰す）
生 tier（rosbag2/ros2_tracing・sub-second）

5. eval_sdk への移送経路（eval_sdk は生 data も ROS も見に行かない）21:82-102,119,189

eval_sdk は純 stdlib コアで rclpy を import せず、ROS 購読を一切持たない（eval_sdk/sink.py:22 / package.xml:16 / CLAUDE.md:5）。生トピックを購読するのは別のドメイン観測ノードで、eval_sdk はそれが渡す観測値を数学・emit するライブラリ。

① 生 ROS topic（near_collision / /bot{n}/odom / State Cache）
sub-second の生信号。④生 tier（rosbag2）にも同時に流れる。21:82-86
② ドメイン観測ノード（観測専用 subscriber / deadlock detector node）
生を count / derive（near_collision を count 化＋inter-robot 距離をログ）。Emergency Guardian の enforcement とは別ノード（観測≠制御・fail-closed と混ぜない）。21:189,190,193
③ eval_sdk.stats（純関数の数学）
count → rate / histogram / percentile / distance。ROS も domain も知らない。eval_sdk/stats.py:1-9 / 21:117
④ eval_sdk.sink（fail-open emit）
score を Langfuse v4 create_score で送信。creds 無/通信断でも no-op（走行無傷）。eval_sdk/sink.py:99-145
⑤ Langfuse（①判断 tier に着地）
同じ trace_id で判断と結合。21:98-101
「観測専用の複製イベントを見に行く」の正確な意味: eval が読むのは near_collision の count 等の観測専用の複製だけで、enforcement 経路には触れない（21:119）。ただし eval_sdk 自身が subscribe するのではなく、ドメイン観測ノードが生から複製・集計して eval_sdk に渡す。Emergency Guardian / collision_monitor / 0.3 m/s は fail-closed の能動制御で、producer ではなく「観測タップ」を指す。
