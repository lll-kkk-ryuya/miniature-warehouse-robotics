2. 排他制御 3層（A + B-3 + C）

同時発火・stale・replay を3層で塞ぐ。この3層は Bridge-owned の安全核で Hermes へ移譲しない。doc08:161-179 / doc15 §2

A
client-side cancel（asyncio.wait_for(2.5s)）。新サイクルが古い LLM 呼び出しを cancel。明示 /v1/runs/{id}/stop は #54 で撤回（stateless chat/completions + in-process dispatch にサーバー側 run 無し）。安全主担保は B-3+C。scheduler.py:309 / doc08:140,173-179
B-3
同世代ガード。gen_store.set で世代公開 → action_map が全 ToolCall に gen_id 注入 → MCP GenChecker が stale gen を reject。gen_store / action_map.py / gen_check.py GenChecker.check
C
replay ガード。action_map が per-call idempotency_key=str(uuid.uuid4()) を mint → MCP check_and_add で同一 call の replay を reject。key は Bridge mint・LLM 由来不可（"minted here, NOT read from LLM-controlled CommandItem.idempotency_key"）。action_map.py:5-9,43 / gen_check.py
安全の非対称性: LLM に「一意で繰り返さない UUID を出せ」と信頼しない。idempotency_key は Bridge が send 時に mint する（action_map.py:7-9）。これが「Hermes に寄せない」核心（box の真ん中を Hermes 線で割らない）。

5. Mode X-ER 詳細図（ER sub-box + L3 Planning Core が実体化）

利点: X-ER 設計を壊しにくい・安全な cross-check・低 confidence / disagreement を operator clarification へ戻しやすい。　弱点: VLA の action 能力をあまり使わない・ER と VLA の disagreement policy が必要。mode-x-er-vla/01 §Option A
B
ER Planner + VLA L3 Candidate Generator。ER が high-level task graph、VLA が Visual Resolver / action candidate を補助。
ER: voice+image+state → task_graph/intent　＋　VLA: image+state+task_graph → grounded target / action candidate
→ Validator → Safety Compiler → Command candidate → MCP / Policy Gate
利点: ER=自然言語/高レベル task・VLA=視覚 grounding/action の役割分担・L3 の一部代行というユーザー意図に近い・action 寄り output も扱いやすい。　弱点: Safety Compiler の設計が要る・VLA output を既存 Command へ落とす境界を慎重に。mode-x-er-vla/01 §Option B
C
Sim-First ER+VLA Policy Evaluation。実機 command path 接続前に Isaac Sim / offline fixture で policy candidate として評価。
recorded/sim image + text + fake state → ER+VLA → integrated candidate → simulator-only evaluator → score / failure analysis → later: Fusion Validator / Safety Compiler
利点: 実機前に相互作用を観察・OpenVLA runtime/GPU/latency 検証・失敗を golden fixture 化。　弱点: 実機 E2E まで遠い・Isaac Sim の scene/camera/object labels/robot model 整備が前提。mode-x-er-vla/01 §Option C
初期判断: Option C → A → B の順。まず sim/offline で input-output を観察 → grounding 補助に使えれば A → action candidate 生成に使えれば B。どの option でも ER/VLA から直接 Nav2 / ROS / Jetson / ESP32 を叩かせない。mode-x-er-vla/01 §初期判断 / README §初期採用判断

6.2 把持/配置 subtask の制御フロー（到着判定は ER でなく L3）

例:「赤い箱へ行って、赤い箱の中の赤いボールを掴んで、トレーに置いて」。ER が高レベル task_graph（navigation + manipulation の kind）を提案し、VLA subtask は L3 が到着・前提を確認した後にだけ起動する。mode-x-er-vla/04 §典型フロー

ER task_graph: t1 navigation/navigate red_box_area → t2 manipulation/grasp red_ball after t1 → t3 navigation/navigate tray_area after t2 → t4 manipulation/place tray after t3
※ command_action=既存 Command へ compile する移動系のみ。skill(grasp/place)=Mode X-ER-VLA 内部 subtask で既存 Command 契約へ入れない。

6.5 SafetyCompiler（manipulation の compile 境界）

既存 Command は navigate / wait / stop / yield / charge の移動命令中心。grasp / place などの manipulation は別の compile 境界が要る。SafetyCompiler は: VLA output class を分類し未対応型を reject / 対象物が ER task graph と一致するか確認 / container・workspace・robot pose の前提を確認 / base motion 禁止 subtask で base motion 相当出力が無いか確認 / trajectory・velocity を直接実行しない / 抽象 skill call へ変換 / result を ManipulationSkillResult として返す。
VlaActionCandidate → SafetyCompiler → ManipulationSkillCall → skill executor / arm controller → ManipulationSkillResult → Task Graph Executor
OpenVLA → controller 直結は禁止。ManipulationSkillCall の package / topic / REST 表現は未凍結。mode-x-er-vla/04 §SafetyCompiler

6.6 task 状態遷移と VLA 起動条件

pending → ready → dispatching → running → succeeded / failed / blocked / cancelled
移動 task
ready → Command Compiler → MCP / Policy Gate → Nav2 → Nav2 result + State Cache → succeeded / failed
VLA task
ready → VlaSubtaskRequest → VLA Adapter → VlaActionCandidate → SafetyCompiler → skill executor → ManipulationSkillResult → succeeded / failed / blocked
VLA subtask 起動条件（すべて満たす）: after 依存が満たされる・State Cache / Nav2 result / task result で前段完了を確認・robot が expected location にいる・emergency active でない・output class が fixture / sim で分類済み・reject できる deterministic validator または SafetyCompiler がある。移動のみの task では VLA を起動しない。mode-x-er-vla/04 §状態遷移 / 03 §VLA 起動前の L3 条件

6.8 Simulation / Safety Gates（G0–G7）と実機前の禁止

Gate内容
G0 fixturerecorded image + text + fake state を ER/VLA に入力し response を保存
G1 output classificationVLA response を grounding / action / trajectory / unknown に分類
G2 fusion decisionER と VLA の統合方式（Cross-check / L3 candidate / Safety Compiler）を選ぶ
G3 offline validatorinvalid robot/action/target/stale/emergency を 0 dispatch（reject reason を残す）
G4 sim-only replayIsaac Sim / offline evaluator で candidate を replay（実機なしで failure 再現）
G5 compiler decisionCommand Compiler で足りるか SafetyCompiler が要るか判断
G6 MCP pathcommand candidate が MCP / Policy Gate を通る（bypass なし）
G7 robot-gated実機接続を人間 gate（Jetson / ESP32 / safety 確認後のみ）

Isaac Sim で見る観点: overhead camera と robot-mounted(wrist) camera のどちらを VLA 入力にするか / ER の object detection・task graph と VLA grounding の一致 / sim の object label と real camera の見え方の差分 / latency が task cycle に収まるか / ER-VLA disagreement の頻度 / failure case を golden fixture 化。mode-x-er-vla/03 §Isaac Sim で見る観点

実機接続前に禁止: VLA output velocity を /cmd_vel へ直接 publish / trajectory を Nav2 action へ直接送る / Jetson の service・REST を ER/VLA へ教える / ESP32 clamp 前提で上位安全を省略 / fixture・sim 未検証の output shape を実機に流す。mode-x-er-vla/03 §Gate・§実機接続前に禁止すること

6.9 OpenVLA 採用判断（未検証・defer されうる）

採用可否そのものが未検証（README §未凍結事項 第1項）。調査項目: model availability / license（商用利用）/ input・output shape / ER integration / robot embodiment（minicar との適合）/ runtime（GPU memory・latency・quantization → Jetson / RunPod 可否）/ simulation / safety / observability。spike 完了条件（offline で response を得る・output 型を説明・ER と VLA を突合・代行できる L3 component を判断・runtime/GPU を記録・license 確認・MCP/Policy Gate を bypass しない接続案）を満たすまで採用しない。mode-x-er-vla/02 §調査項目・§初期 spike の完了条件 / README §未凍結事項

OpenVLA を使う / 使わない: 移動が named location で完結・Nav2/Open-RMF で解ける・把持/配置不要なら 使わない（Mode X-ER）。移動後の grasp / place / align / dock が必要・対象の位置姿勢が毎回変わる・output を grounding_report / action_candidate に分類でき reject path が sim で確認済みなら 候補。mode-x-er-vla/04 §OpenVLA を使わない判断・§OpenVLA を使う判断 / 02 §調査後の分岐
