eval_sdk 拡張 詳細図 — core（再利用）vs plugin（差替）の3分割

新しい現場・新しい指標を足すとき、eval_sdk の数式（stats）は書き換えず、ドメイン側に「producer ＋ score 名」を足して既存 stats/sink を再利用する。何が core（汎用・一度作れば再利用）か、何が plugin（現場ごとに差し替える）かを 3 つに割るのが拡張の肝。「指標定義・データ producer はドメインに残す」（21:75）が原則。正本 architecture/21 ＋ 03 / 05。

正本: 21（指標定義はドメイン 21:75・5 モジュール 21:129-139・name registry は Phase 2 contract 21:244・Mode A 指標 判定者 21:318-326）／ 03（stage ごと 再利用 core / 差し替え plugin 03:224-235）／ 05（L3 Validator の core/案件/既存 tool 05:128-169）。 区分: eval_sdk.stats（数式 core）は Phase 1 実装済（eval_sdk/stats.py）／ ScoreSpec 名 registry は Phase 2 contract（未凍結・21:244）／ 指標定義・producer は ドメイン永続。新 frozen 契約 / 指標 / しきい値を発明しない。製造例は 03/05 の literal を図示。

0. なぜ「3 分割」か — 数式は共通、意味はドメイン

「合格率を出す割り算」はどの業界でも同じ＝core。だが「何を合格とするか」「その pass/fail 信号を誰が出すか」は現場ごとに違う＝plugin。eval_sdk は前者（数学・emit・join-key の素材）だけを持ち、後者（指標定義・producer）はドメインに残す。21:75

core（汎用・一度作れば再利用・★eval_sdk）
plugin（現場ごとに差し替える・ドメイン）
manifest（名前語彙・Phase 2 で contract 化）

1. 核心 — 何が core / plugin / manifest か21:75,129-139,244

① 数式（core）再利用
eval_sdk.stats（純関数・Phase 1 実装済）
  • percentile / path_lengths / distance_traveled（既存純関数）
  • pass/total は単純な割り算（rate は doc21 §6 の汎用 helper 案・現状 stats 未実装＝domain が合成）21:178
  • emit = eval_sdk.sink（fail-open）
  • join-key = eval_sdk.seed（決定的 trace_id）
② 定義＋producer（plugin）差替
ドメイン（現場ごと・永続的にドメイン）
  • 何を success とするか（goal 述語・しきい値）
  • その pass/fail 信号を出す producer
  • 例: 「挿入トルク検査が閾値内か」を出すノード
  • 設備 registry・工程 allowlist・interlock policy
③ 名前（manifest）contract
dashboard 列見出し（Phase 2 で contract）
  • score 名 = insertion_torque_pass_rate 等
  • nav 標準名（success_rate/spl …）は plugin="nav" 登録
  • 改名は履歴孤児化＝contract ラベル＋予告
  • 現状は SCORE_* 予約（inert）21:244
増やすとき書くのは ② と ③ だけ。① の数式は書き換えない（既存 sink ＋ 純関数 percentile/path_lengths/distance_traveled を再利用。pass/total の rate は doc21 §6 helper 案・現状 stats 未実装＝domain が合成）。＝「製造現場が要件ごとに指標を増やす」とは 新 producer ＋ 新 score 名を足すこと。これが eval_sdk が 産業非依存のまま再利用できる正体。21:75,118

3. 「指標を 1 個増やす」手順（new producer ＋ score 名 ＋ 既存再利用）

① success を定義（plugin）
「何を pass とするか」の述語・しきい値を現場仕様として決める。05:149-157 / 21:390-392
② producer を書く（plugin）
その pass/fail（または生信号）を出すドメインノード。観測専用レーン（enforcement と混ぜない）。21:195
③ 既存 stats で数値化（core・書かない）
既存純関数（percentile / path_lengths / distance_traveled）をそのまま呼ぶ＝新数式を書かない。pass/total は単純な割り算（rate helper は doc21 §6 案・現状 stats 未実装）。eval_sdk/stats.py / 21:178
④ sink で emit（core・書かない）
eval_sdk.sink.score(trace_id, name, value, …) を fail-open 送信。同じ trace_id で他指標と束ねる。eval_sdk/sink.py:99-136
⑤ score 名を manifest 登録（Phase 2・contract）
dashboard 列見出しを固定（改名は履歴孤児化＝contract ＋全レーン予告）。21:244
手で書くのは ① success 定義と ② producer だけ。③④ は既存 core をそのまま使う。⑤ は Phase 2 の name contract。＝「stats 数式の追加」は基本不要。

4. L3 Validator の core / 案件 / 既存 tool（05 を図示）05:128-169

L3 Validator は最も案件固有の作り込みが大きい box。05 は「再利用 core / 案件ごとに作る / 既存 tool で済む」を明示している。

再利用 core汎用
JSON parse・schema/required field validation・stable error code・validation report 生成・DAG/task graph 検証・fixture replay runner・audit event 生成。05:139-147
案件ごとに作る差替
known target/robot rule・allowed action・confidence threshold・camera calibration/homography/snap rule・operator clarification rule・task graph policy・unsafe output catalog。05:149-157
既存 tool で済む乗る
Pydantic/JSON Schema（schema validation）・NetworkX（task graph）・BehaviorTree.CPP/Nav2 BT（task logic 参考）・OpenCV（visual geometry）・pytest/golden fixture（replay/regression）。05:159-168
境界: 「この顧客現場で赤箱を shelf_A に snap してよいか」「低 confidence で operator に聞くか reject するか」は既存 tool に無い＝商用案件の大枠 rule 設計＝plugin の核。05:169
