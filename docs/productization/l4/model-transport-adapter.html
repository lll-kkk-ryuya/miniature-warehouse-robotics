Hermes は {llm, er, vla, stt} の 既定 (default) transport (F1)。現行 LLM は model: hermes-agent で実装済み。ER / VLA / STT / Vision も既定は Hermes。direct / worker は 明示的 fallback（Hermes が modality / runtime / response shape / latency を満たせない、または必要な fork が未配備の場合のみ）。従来「第一候補」と書いていた方針を default として明示化したもので、新しい config key / contract は増やさない。
hermes default
OpenAI互換 API、provider routing / fallback、vision / STT / MCP entry。server-side motion 実行は使わない。ER native audio は input_audio fork が要る（下記 §0.5 / F3）。
direct explicit fallback
Hermes 非対応 API / response shape / latency 要件の fallback。ER native audio の 現行経路（fork が ship されるまで・F4）。
worker explicit fallback
GPU / separate process runtime。VLA や重い local model の fallback。
▼
L3 Handoff seam
どの transport でも同じ内部 shape へ正規化して L3 Planning Core へ渡す。
▼
action_map / MCP / Policy Gate seam
Bridge が gen_id / idempotency_key を mint。motion 採用判定は Hermes に渡さない。

0.5 ER native audio と input_audio fork

audio modality の carve-out（measured 2026-06-27）: ER の image / text は unforked Hermes 経由で動作する（PROBE-3, HTTP 200・F2）。一方 ER の native audio は、unforked Hermes の /v1/chat/completions に input_audio を渡すと HTTP 400 unsupported_content_type になる（PROBE-2, 2026-06-27 実測・F2）。
fork demonstration（2026-06-27・demonstrated, not-yet-shipped）F3
native audio を Hermes に載せるには hermes-agent v0.15.1 への 2-file fork が要る:
• patch 1: gateway/platforms/api_server.py — _normalize_multimodal_content が input_audio を受理、_content_has_visible_payload と新設 _AUDIO_PART_TYPES を追加。
• patch 2: agent/gemini_native_adapter.py — _extract_multimodal_parts の input_audio を Gemini inlineData{mimeType: audio/wav} へ写像。
LIVE RESULT: HTTP 200、ER が native audio を理解（音声にしか無い語の transcript を返した）、latency は direct と comparable（lean Hermes median 3.69s vs direct 4.24s, n=4・ER-thinking confound あり）、prompt token は +~408/call。
TARGET と CURRENT を取り違えない（F4）: TARGET は audio を含む全 modality で Hermes default（audio は上記 fork 経由）。ただし fork はまだ productionize / ship していないため、CURRENT の audio path は direct であり、これは default 化後も permanent fallback として残す。「audio は今 Hermes default」とは書かない — fork が出荷されて初めて default になる。

領域Hermes-managedBridge-owned

STT / VisionHermes が扱える provider / plugin / image input entryaudio / image / state / calibration の final bundle、artifact retention

Trace (Langfuse = Opt C)FUTURE TARGET = Hermes-owned plugin。HLF-G0〜G5 を全て満たした場合に 限って 切替（gate 越えが前提・F6）CURRENT = Bridge-owned root trace（create_trace_id(seed=run_id:gen_id)・managed-prompt link・4-provider 単一 codepath で公平比較）。Hermes plugin は double-counting 回避のため disabled。flip 後も eval_sdk が outcome score（result/SR/SPL/collision/deadlock）を所有し続ける（Hermes は robot result を見ない・F6）
provider routing の含意（F5）: Hermes は server-side で active model 1 つを持ち、per-request の provider 選択は無い。よって Mode A/B/C の 4-provider 比較は request field では切替えられず、provider ごとに別 gateway（config + restart）を立てる必要がある。これは新しい contract / request field を増やすものではない。

2. 商用再利用で保管するもの

汎用化するもの
adapter interface、transport selection、request / response envelope、raw output recorder、golden fixture、trace metadata、provider latency / cost hint。
site ごとに変えるもの
provider、model、region、quota、auth、calibration、known locations、operator policy、artifact retention、redaction rule。
安全核: model output は提案であり、Nav2 / Open-RMF / micro-ROS / /cmd_vel を直接呼ばない。timeout / provider error / malformed response は 0 dispatch。
