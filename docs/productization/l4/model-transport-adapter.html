Hermes は第一候補 transport。現行 LLM は model: hermes-agent で実装済み。ER / VLA / STT / Vision は Hermes-first で追加する。
hermes
OpenAI互換 API、provider routing / fallback、vision / STT / MCP entry。server-side motion 実行は使わない。
direct
Hermes 非対応 API / response shape / latency 要件の fallback。
worker
GPU / separate process runtime。VLA や重い local model の fallback。

1. Hermes-managed と Bridge-owned の分担

領域Hermes-managedBridge-owned
Model transportOpenAI互換 API、provider routing / fallback、custom endpointrequest_id、timeout、0 dispatch、L3 Handoff 正規化
STT / VisionHermes が扱える provider / plugin / image input entryaudio / image / state / calibration の final bundle、artifact retention
MCPtool filter、read-only / operator-facing tool の接続候補motion tool の accepted path、Policy Gate、idempotency_key
TraceHermes plugin は HLF-G0〜G5 後に再評価現採用は Bridge-owned root trace。二重 generation を避ける

2. 商用再利用で保管するもの

汎用化するもの
adapter interface、transport selection、request / response envelope、raw output recorder、golden fixture、trace metadata、provider latency / cost hint。
site ごとに変えるもの
provider、model、region、quota、auth、calibration、known locations、operator policy、artifact retention、redaction rule。
安全核: model output は提案であり、Nav2 / Open-RMF / micro-ROS / /cmd_vel を直接呼ばない。timeout / provider error / malformed response は 0 dispatch。

3. acceptance gate

Gate内容
L4M-G0既存 commander LLM の Hermes regression を壊さない。
L4M-G1provider_type と transport を直交 field として fixture 化する。
L4M-G2Hermes / direct / worker が同じ L3 Handoff input に正規化される。
L4M-G3timeout / provider_error / malformed_response / empty_output / unsupported_modality は 0 dispatch。
L4M-G4raw output、adapter report、latency、transport、provider metadata が artifact 化される。
L4M-G5provider fallback は fixed-provider comparison run と混在しない。
L4M-G6Hermes Langfuse plugin 採用時は HLF-G0〜G5 を全て満たし、二重 generation がない。
