0. なぜ再利用できるか（Core 不変 / Plugin 差替の総括）

L3 Planning Core は「AI model の曖昧な出力」と「ロボットの安全な実行契約」の間に立つ deterministic な正規化・検証層で、この役割は倉庫固有ではない。倉庫デモは最初の 1 事例にすぎず、Core（産業非依存）は変えず・Plugin（現場差分）だけを差し替えるのが商用・産業横断の差替軸。以下の 6 資産はすべてこの 1 つの分離の派生。03:218-220

6. ISO / IATF 16949 — 「動機」と「機能安全の境界」03:235,245-251

ISO/IATF は repo 内に 契約・schema・しきい値としての定義は無く、設計 docs が ③の商用価値の動機と L3 が機能安全を置換しない境界として言及するのみ。

動機（なぜ ③ が刺さるか）03:235
IATF 16949（自動車部品サプライヤー向け品質規格・ISO 9001 ベース）/ ISO 9001（一般 QMS）は「どの判断がどの actuation を生んだか」の追跡可能性（traceability/genealogy）を要求する。③ audit 1:1 はこの要件にそのまま刺さる＝B2B 製造での競争力。視覚的なものではなく、工程・記録・追跡の仕組みの認証。
境界（L3 が担わないもの）03:247
機能安全（E-stop / light curtain / safety PLC / ISO 10218・ISO/TS 15066・ISO 13849・IEC 61508）は認証済み safety hardware（L0/L1 相当）に残す。Validator の emergency policy plugin は safety state を読んで拒否するだけで safety system を置換しない。
準拠主張をしない: 本デモ（YouTube）で ISO/IATF を取得する話ではない。これらは「audit 1:1 や stable error code が、将来の製造案件でこういう規格要件に効く」という動機・境界としての引用のみ。横展開しても L3 自身は実行許可を持たず、最後は必ずその現場の安全経路に渡せる command 候補で終わる点は不変。03:252
